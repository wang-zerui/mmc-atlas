# CriticGPT / LLM Critics Help Catch LLM Bugs ## 一句话 - 一个经 RLHF 训练的 critic 模型为代码撰写自然语言评述，训练员更偏好这些评述而非人类评述，且能发现更多 bug；人类+critic 组队的幻觉比 critic 单独工作时更少。 ## 问题 - 人类评估者无法可靠地发现复杂的模型生成代码中的 bug，导致 RLHF 反馈质量下降（这是一个可扩展监督问题）。 ## 洞见 - 一个专门经 RLHF 训练来评述代码的模型，能够用自然语言指出真实的 bug，而人类+critic 组队的表现优于二者单独工作，从而验证了可扩展监督的可行性。 ## 方法 - CriticGPT 通过 RLHF 训练，为 ChatGPT 生成的代码撰写自然语言评述；评估在人工插入的 bug 和自然出现的 bug 上进行，并考察人类+critic 组队的场景。 ## 设置 - 对 critic 评述与人类评述进行人类偏好比较，测量相对于付费代码审查员的 bug 检出率，以及在有无人类辅助情况下的精确率/召回率权衡。 ## 结果 - 在自然出现的 bug 上，训练员有 63% 的情况更偏好 CriticGPT 的评述，且 critic 比付费人类审查员发现更多 bug；人类+critic 组队的幻觉比 critic 单独工作时更少（已在摘要中确认）。 ## 挑战 - critic 可能会幻觉出并不存在的问题或进行吹毛求疵；如何在召回真实 bug 与减少幻觉 bug 之间取得平衡是核心问题。 ## 局限 - 研究聚焦于相对较短、仅含单个 bug 的代码片段；critic 仍会产生幻觉，可能难以扩展到大型、分散的真实世界缺陷。

背景

RLHF 依赖人类训练员评估模型输出，但当模型处理代码等更难的任务时，人类越来越容易漏掉细微错误，从而威胁到奖励信号的可靠性。

问题

人类评估者无法可靠地发现复杂的模型生成代码中的 bug，导致 RLHF 反馈质量下降（这是一个可扩展监督问题）。

动机 / 分析

用一个能揭示缺陷的 AI critic 来辅助人类训练员，可以在模型能力超越无辅助人类判断时，仍让人类监督保持有效。

关键洞见

一个专门经 RLHF 训练来评述代码的模型，能够用自然语言指出真实的 bug，而人类+critic 组队的表现优于二者单独工作，从而验证了可扩展监督的可行性。

挑战

critic 可能会幻觉出并不存在的问题或进行吹毛求疵；如何在召回真实 bug 与减少幻觉 bug 之间取得平衡是核心问题。

方法

CriticGPT 通过 RLHF 训练，为 ChatGPT 生成的代码撰写自然语言评述；评估在人工插入的 bug 和自然出现的 bug 上进行，并考察人类+critic 组队的场景。

实验设置

对 critic 评述与人类评述进行人类偏好比较，测量相对于付费代码审查员的 bug 检出率，以及在有无人类辅助情况下的精确率/召回率权衡。

结果

在自然出现的 bug 上，训练员有 63% 的情况更偏好 CriticGPT 的评述，且 critic 比付费人类审查员发现更多 bug；人类+critic 组队的幻觉比 critic 单独工作时更少（已在摘要中确认）。

局限

研究聚焦于相对较短、仅含单个 bug 的代码片段；critic 仍会产生幻觉，可能难以扩展到大型、分散的真实世界缺陷。

与领域的关系

一种用于可扩展监督的评述生成型 verifier；在概念上与 Generative Verifiers 和 Agent-as-a-Judge 相邻，后两者同样产出有推理依据的验证而非标量分数。
