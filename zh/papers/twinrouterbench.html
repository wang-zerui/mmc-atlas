# TwinRouterBench ## 一句话 - 面向长程 agentic 应用的 step-level routing 基准，包含一个静态确定性赛道（970 个 router 可见的前缀）和一个在 SWE-bench Verified 上实时运行 router 的动态赛道，用于衡量真实任务解决率与 API 花费。 ## 问题 - 现有 routing 基准只评测单条 query，无法衡量 step-level routing 如何影响端到端 agentic 任务的成功率与成本。 ## 洞见 - 将确定性回放赛道（固定前缀）与实时执行赛道配对，既能让 router 可复现地对比，也能在真实任务解决率与花费上进行衡量。 ## 方法 - 两条赛道：静态确定性赛道包含从 SWE-bench 和 BFCL 抽取的 970 个 router 可见前缀；实时动态赛道让 router 在 SWE-bench Verified（520 个实例）上端到端运行，衡量任务解决率与 API 美元花费。 ## 设置 - 静态赛道：970 个前缀；实时赛道：SWE-bench Verified，含 520 个实例，衡量候选 router 的真实解决率与花费。 ## 结果 - 确认了双赛道的 agentic step-level 设计、970 个前缀 / 520 个实例，以及实时 SWE-bench Verified 花费衡量（在 claim_checked 中已核实）；v1 2026 年 5 月 14 日（v2 5 月 22 日），未声明发表 venue。 ## 挑战 - 从 SWE-bench/BFCL 构建 970 个 router 可见的 step 前缀，并编排一个让 router 在 SWE-bench Verified 的真实智能体内部运行的实时赛道。 ## 局限 - 实时赛道结果取决于所用的特定智能体与模型池；为无 venue 的预印本。

背景

Agentic 应用（编程、研究智能体）在一条长轨迹上会发起大量模型调用，因此 routing 决策发生在 step 层面，而非针对单条独立 query。

问题

现有 routing 基准只评测单条 query，无法衡量 step-level routing 如何影响端到端 agentic 任务的成功率与成本。

动机 / 分析

要公平评测 agentic router，既需要可复现的静态对比，也需要一个能刻画真实任务上逐步决策累积效应的实时环境。

关键洞见

将确定性回放赛道（固定前缀）与实时执行赛道配对，既能让 router 可复现地对比，也能在真实任务解决率与花费上进行衡量。

挑战

从 SWE-bench/BFCL 构建 970 个 router 可见的 step 前缀，并编排一个让 router 在 SWE-bench Verified 的真实智能体内部运行的实时赛道。

方法

两条赛道：静态确定性赛道包含从 SWE-bench 和 BFCL 抽取的 970 个 router 可见前缀；实时动态赛道让 router 在 SWE-bench Verified（520 个实例）上端到端运行，衡量任务解决率与 API 美元花费。

实验设置

静态赛道：970 个前缀；实时赛道：SWE-bench Verified，含 520 个实例，衡量候选 router 的真实解决率与花费。

结果

确认了双赛道的 agentic step-level 设计、970 个前缀 / 520 个实例，以及实时 SWE-bench Verified 花费衡量（在 claim_checked 中已核实）；v1 2026 年 5 月 14 日（v2 5 月 22 日），未声明发表 venue。

局限

实时赛道结果取决于所用的特定智能体与模型池；为无 venue 的预印本。

与领域的关系

将 routing 基准测试（RouterBench M028、LLMRouterBench M122）扩展到 agentic step-level 场景，与 Switchcraft（M124）等 agentic router 相呼应。
