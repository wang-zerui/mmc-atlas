# LLMRouterBench ## 一句话 - 一个大规模的统一路由 benchmark（400K+ 实例、21 个数据集、33 个模型、10 个 baseline），其核心发现是许多近期及商用 router 都无法可靠地超越一个简单 baseline，而且 router 的骨干 embedding 几乎无关紧要。 ## 问题 - 缺少一个大规模、统一的 benchmark 来公平比较各 router，并检验它们是否真的能胜过平凡的 baseline。 ## 洞见 - 在大规模下、对照一个简单 baseline 和一个 oracle 评测时，许多复杂 router 几乎没有真正的优势，而 embedding 骨干的选择贡献甚微。 ## 方法 - 一个统一的 benchmark harness，预先计算了 33 个模型在 400K+ 实例、21 个数据集上的输出；针对一个简单 baseline 和一个 oracle 上界评测 10 个路由 baseline 加商用 router，并对 embedding 骨干做消融。 ## 设置 - 400K+ 实例、21 个数据集、33 个模型、10 个 baseline，加上商用 router；在成本-质量前沿上相对简单 baseline 和 oracle 给出指标。 ## 结果 - 确认了 400K+ 实例 / 33 个模型 / 10 个 baseline 的规模，以及以下发现：router 无法可靠地击败简单 baseline、与 Oracle 之间存在持续差距、骨干 embedding 的选择影响甚微（已在 claim_checked 中核实）。 ## 挑战 - 在 21 个数据集、33 个模型上汇集 400K+ 实例，并一致地实现 10 个 router baseline 以做公平比较。 ## 局限 - 与 RouterBench 一样，离线/静态评测无法涵盖 agentic 或多轮路由；为预印本，未声称 venue。

背景

LLM router 大量涌现且各有相互竞争的宣称，但评测使用的模型池和数据集不一致，使进展难以判断。

问题

缺少一个大规模、统一的 benchmark 来公平比较各 router，并检验它们是否真的能胜过平凡的 baseline。

动机 / 分析

严谨的科学比较需要标准化的规模和强 baseline；以往的 benchmark（如 RouterBench 2024）规模较小，给出的图景过于乐观。

关键洞见

在大规模下、对照一个简单 baseline 和一个 oracle 评测时，许多复杂 router 几乎没有真正的优势，而 embedding 骨干的选择贡献甚微。

挑战

在 21 个数据集、33 个模型上汇集 400K+ 实例，并一致地实现 10 个 router baseline 以做公平比较。

方法

一个统一的 benchmark harness，预先计算了 33 个模型在 400K+ 实例、21 个数据集上的输出；针对一个简单 baseline 和一个 oracle 上界评测 10 个路由 baseline 加商用 router，并对 embedding 骨干做消融。

实验设置

400K+ 实例、21 个数据集、33 个模型、10 个 baseline，加上商用 router；在成本-质量前沿上相对简单 baseline 和 oracle 给出指标。

结果

确认了 400K+ 实例 / 33 个模型 / 10 个 baseline 的规模，以及以下发现：router 无法可靠地击败简单 baseline、与 Oracle 之间存在持续差距、骨干 embedding 的选择影响甚微（已在 claim_checked 中核实）。

局限

与 RouterBench 一样，离线/静态评测无法涵盖 agentic 或多轮路由；为预印本，未声称 venue。

与领域的关系

是 RouterBench (M028) 更大规模、更具批判性的后继；其发人深省的结果推动了 agentic benchmark（TwinRouterBench M127）以及该家族中更好的 router 设计。
