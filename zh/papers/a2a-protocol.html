# A2A ## 一句话 - A2A 是一个开放的点对点 protocol，让基于不同 framework、由不同厂商构建的 AI agent 能够相互发现、委派任务并共享结果，而无需暴露各自的内部 memory、工具或逻辑。 ## 问题 - 此前没有一种通用语言让一个自主 agent 能够声明自己的能力，并在保持双方不透明性和安全性的前提下将任务移交给另一个 agent。 ## 洞见 - 将每个 agent 视为一个不透明的服务，它发布机器可读的能力描述并通过既定的生命周期接受任务，从而在保持内部状态私密的同时实现安全委派。 ## 方法 - agent 发布 Agent Cards（在 /.well-known/agent-card.json 提供）描述其技能；对等方通过 HTTP 使用 JSON-RPC 2.0 和 Server-Sent Events 进行流式通信，采用 OAuth2/标准认证，并通过既定的任务生命周期驱动任务。 ## 设置 - 通过生态系统采用而非 benchmark 来验证；发布时即有 50+ 合作伙伴，并演示了跨 framework 的 agent 互操作性。 ## 结果 - 没有性能数据；已验证的事实包括：2025 年 4 月 9 日发布，拥有 50+ 合作伙伴，2025 年 6 月捐赠给 Linux Foundation（150+ 组织），Agent Cards 位于 /.well-known/agent-card.json，并明确与 MCP 互补（agent 到… ## 挑战 - 在不泄露 agent 内部状态的前提下，支持长时间运行和流式任务、安全认证、能力发现以及与 framework 无关的互操作性。 ## 局限 - A2A 管理的是 agent 间的消息传递和任务委派，但不标准化 agent 如何访问自身的工具/数据（那是 MCP 的职责），也不规定内部推理或 memory。

背景

企业越来越多地运行构建于异构技术栈（LangGraph、CrewAI、Semantic Kernel）上的 agent，这些 agent 需要跨组织和跨 framework 边界进行协作。

问题

此前没有一种通用语言让一个自主 agent 能够声明自己的能力，并在保持双方不透明性和安全性的前提下将任务移交给另一个 agent。

动机 / 分析

需要一个 agent 到 agent 协作的标准，以支持复杂的跨厂商工作流，同时不强迫 agent 共享专有内部实现或统一到单一 framework。

关键洞见

将每个 agent 视为一个不透明的服务，它发布机器可读的能力描述并通过既定的生命周期接受任务，从而在保持内部状态私密的同时实现安全委派。

挑战

在不泄露 agent 内部状态的前提下，支持长时间运行和流式任务、安全认证、能力发现以及与 framework 无关的互操作性。

方法

agent 发布 Agent Cards（在 /.well-known/agent-card.json 提供）描述其技能；对等方通过 HTTP 使用 JSON-RPC 2.0 和 Server-Sent Events 进行流式通信，采用 OAuth2/标准认证，并通过既定的任务生命周期驱动任务。

实验设置

通过生态系统采用而非 benchmark 来验证；发布时即有 50+ 合作伙伴，并演示了跨 framework 的 agent 互操作性。

结果

没有性能数据；已验证的事实包括：2025 年 4 月 9 日发布，拥有 50+ 合作伙伴，2025 年 6 月捐赠给 Linux Foundation（150+ 组织），Agent Cards 位于 /.well-known/agent-card.json，并明确与 MCP 互补（agent 到 agent 对比 agent 到工具）。

局限

A2A 管理的是 agent 间的消息传递和任务委派，但不标准化 agent 如何访问自身的工具/数据（那是 MCP 的职责），也不规定内部推理或 memory。

与领域的关系

与 MCP 互补：MCP 赋予 agent 工具，A2A 让该 agent 与对等方协作；在分层互操作性路线图中位于 MCP 之上，并在 agent 协议综述和 ProtocolBench 中与 ACP/ANP 进行对比。
