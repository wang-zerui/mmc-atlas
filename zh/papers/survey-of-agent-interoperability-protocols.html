背景

由 LLM 驱动的自主 agent 必须连接工具、共享上下文并跨系统协调，但临时性的集成难以规模化、难以保障安全，也难以跨领域泛化。

问题

多个相互竞争的 agent 通信标准已经出现，它们的适用范围相互重叠且不清晰，使得实践者缺乏在何时选用哪一个的指导。

动机 / 分析

需要一个结构化的对比，以厘清每种 protocol 的优势和部署定位，并描绘出它们如何组合成一个连贯的技术栈。

关键洞见

这四种 protocol 是互补而非竞争关系，可以分层组合成一条从工具访问到开放去中心化 agent 网络的演进路径。

挑战

将采用不同传输层、身份模型和发现机制的 protocol 调和进单一的对比框架和采用路径。

方法

系统性地对比 MCP（JSON-RPC 工具调用）、ACP（基于 RESTful HTTP、采用 MIME multipart 消息与代理/DID 身份，源自 KQML/FIPA-ACL 谱系）、A2A（基于能力的 Agent Cards 用于对等任务委派）和 ANP（通过 W3C DIDs 和 JSON-LD 实现开放网络发现），涵盖交互模式、发现、通信模式和安全模型。

实验设置

这是一篇概念性/分析性综述而非实证研究；没有运行 benchmark 实验。

结果

没有定量结果；已验证的贡献是四种 protocol 的对比以及分层采用路线图：MCP（工具）-> ACP（结构化、会话感知的消息传递）-> A2A（协作式任务执行）-> ANP（去中心化的 agent 市场）。

局限

作为一篇综述，它提供的是框架和指导，而非对 protocol 性能或互操作性权衡的实证测量。

与领域的关系

提供了将 MCP、A2A、ACP 和 ANP 联系起来的分类体系；其实证对应工作是 ProtocolBench，并与 ICML 2026 的“Web of Agents”立场论文共享对生态系统碎片化的关切。
